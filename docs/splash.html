01 — The Problem

Fragmented Observability at PayPal Scale

PayPal processes billions of transactions across a global multi-cloud platform. Every checkout, authorization, fraud decision, and settlement flows through interdependent microservices — each generating signals that today live in separate silos.

Current StateOperational Impact
CAL Logging Proprietary schema, early-2000s design No off-the-shelf tooling interoperability; specialist knowledge required to query
Siloed Signals Traces, metrics, logs in separate tools Correlating a merchant complaint to a root cause requires hopping between 3–5 tools
Cardinality Pressure Millions of merchant IDs, transaction IDs, regions Metrics backends choke on high-dimensional label combinations at PayPal's volume
No Trace Context CAL lacks distributed trace propagation Cross-service incident investigation is manual and error-prone
Operational Overhead Separate ingest pipelines per signal type Platform engineering time spent maintaining pipelines, not building capabilities
02 — The Solution

Elastic Serverless + OpenTelemetry: One Platform for All Signals

Elastic Serverless provides a zero-ops observability backend purpose-built for the volume, cardinality, and query complexity that PayPal's platform demands. Combined with OpenTelemetry as the instrumentation standard, it replaces the fragmented tool landscape with a single unified destination.

No Cluster Management

No shard sizing, no JVM tuning, no rolling restarts. Provision a complete Observability project in under 60 seconds.

Native OTLP Ingestion

Services ship logs, metrics, and traces directly via OTLP gRPC or HTTP. No Collector required in the data path.

Unified Correlation

Every signal shares trace.id, merchant.id, and transaction.id — one field pivots a merchant complaint to full telemetry.

Built-in AI Assistant

Every project includes an Elastic AI Assistant with live access to your telemetry — no data export, no separate AI tooling.

Cardinality at Scale

Elastic's data tier architecture handles high-dimensional metric label combinations that overwhelm traditional TSDB backends.

Autonomous Remediation

Elastic Workflows trigger automated playbooks — alert → investigate → remediate — without human intervention at 3 AM.

03 — CAL Log Replacement

From CAL to OpenTelemetry Semantic Conventions

The goal is not just "logs in a new place." It's a shared schema understood by every service, every tool, and every engineer — no custom parsers, no proprietary adapters.

CAL FieldOTel Semantic ConventionValue Gained
CAL_TYPEservice.nameStandard across all CNCF tooling; automatic service map grouping
CAL_MODULEservice.namespaceNative subsystem grouping in Kibana service maps
TXN_IDtransaction.id / trace.idAutomatic distributed trace correlation across all services
MERCHANT_IDmerchant.id (custom attribute)Pivot any signal — log, metric, trace — by merchant in a single field

DURATION_MSduration (span attribute)Native latency percentile histograms with no pre-aggregation
No equivalentdeployment.environment, cloud.regionInfrastructure and deployment context baked into every signal
04 — ES|QL at Merchant Scale

One Query Language Across All Signals — No Splunk SPL, No PromQL

ES|QL is a pipe-based query language purpose-built for high-volume telemetry. Every investigation — from a merchant complaint to a production incident — uses the same syntax across logs, metrics, and traces.

-- Which services are failing for a specific merchant? FROM logs.otel | WHERE merchant.id == "M-98231" AND severity_text == "ERROR" | STATS errors = COUNT(*) BY service.name | SORT errors DESC
-- P99 checkout latency by merchant tier and cloud region FROM metrics-* | WHERE @timestamp > NOW() - 15 MINUTES AND service.name == "checkout-service" | STATS p99 = PERCENTILE(transaction.duration, 99) BY merchant.tier, cloud.region | SORT p99 DESC
-- Trace all service hops for a single transaction FROM traces-* | WHERE transaction.id == "TXN-44821903" | STATS spans = COUNT(*), errors = COUNT_IF(error == true) BY service.name | SORT spans DESC
